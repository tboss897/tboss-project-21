SmartSchool Wallet — Implementation Plan
Overview
Goal: Build a production-grade e-wallet system for secondary schools with:

Web app (Phase 1) → Admin dashboard + Seller/Store dashboard (desktop-optimized)
Mobile apps (Phase 2 — deferred) → Parent app, Student app, Seller app (QR scanner)
Current state: A partially scaffolded Django backend + Vite/React frontend exists. Models, views, serializers, and page stubs are in place but contain structural issues and incomplete features.

IMPORTANT

Why React Native (Expo)? The spec calls for mobile apps for Parent, Seller, and optionally Student. React Native lets us share the same API client code, auth logic, and business rules with the web frontend. Expo simplifies camera access (QR scanning) and push notifications — the two main mobile-specific features.

2. Database: Stick with PostgreSQL
The Docker compose already uses Postgres. The spec says MySQL but since the project is already wired to Postgres via psycopg2-binary and the ORM abstracts the difference, we'll keep Postgres. It's superior for DECIMAL precision, select_for_update(), and JSON fields.

3. Deployment: Fully Managed (No Docker)
No Docker required — everything runs on managed platforms.

Layer	Platform	Details
Frontend	Vercel	Vite/React static site. Auto-deploys from frontend/ on git push. VITE_API_URL env var points to backend URL.
Backend	Railway or Render	Plain Django deployment — just push code, platform handles Python runtime. No Dockerfile needed.
Database	Supabase Postgres	Free tier available. Django connects via DATABASE_URL connection string. Used for both local dev and production.
Redis	Upstash	Serverless Redis. Free tier: 10k commands/day (plenty for rate limiting + Celery). Connected via REDIS_URL.
Task Queue	Celery on Railway	Runs as a separate worker process on the same Railway project.
Local dev workflow:

bash
# Backend (venv + pip — no Docker)
cd backend
python -m venv venv && source venv/bin/activate
pip install -r requirements.txt
cp .env.example .env          # Add Supabase + Upstash URLs
python manage.py runserver    # Django dev server on :8000
# Frontend (Vite)
cd frontend
npm install && npm run dev    # Vite dev server on :5173
Production deploy:

bash
# Frontend → git push → Vercel auto-deploys
# Backend  → git push → Railway/Render auto-deploys
# Database → Supabase dashboard (always on)
# Redis    → Upstash dashboard (always on)
TIP

The existing docker-compose.yml and Dockerfile files will be kept in the repo as optional — useful if someone wants to run everything locally in containers. But they are not required for development or production.

4. User Model Redesign
The current User model mixes parent/seller/admin fields into one table. The fix:

User (auth)          → email, password, role, is_active
  └── Student        → full_name, matric_no, department, level, pin_hash (linked via FK to User)
  └── ParentProfile  → phone, monitoring_enabled, daily_limit
  └── SellerProfile  → location, service_type, store_status
  └── AdminProfile   → (minimal — just the User record)
  └── ParentStudent  → parent(FK→User), student(FK→Student)  # M2M linking table
WARNING

This is a breaking migration. The existing db.sqlite3 dev database will be wiped and re-migrated. No production data exists yet so this is safe.
